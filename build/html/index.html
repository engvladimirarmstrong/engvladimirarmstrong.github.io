AI Neuron (AIN) Design Under Test (DUT)

The AI Neuron (AIN) SystemVerilog RTL module is Design Under Test (DUT). The AIN is the building block of Artificial Intelligence (AI) neural networks.

AIN Model

AIN Mathematical Model
\[y = f\left( \sum_{i=1}^{n} w_i x_i + b \right)\]

The Artificial Neuron Mathematical Model output \(y\) is calculated using the following steps:

  • Step #1. Weighted Sum: Calculate the sum of inputs multiplied by weights.

  • Step #2. Bias: Add a bias term to the sum.

  • Step #3. Activation: Apply a non-linear activation function:

    • Sigmoid: Outputs a value between 0 and 1.

    • ReLU: Rectified Linear Unit, outputs the input if positive, else zero.

AIN RTL Source Code
 1//----------------------------------------------------------------------------------------------------------------------
 2// Artificial Intelligence Neuron (AIN) RTL Design SystemVerilog Module
 3//----------------------------------------------------------------------------------------------------------------------
 4
 5// This module describes a Neuron (AIN) Design with 2 inputs and 4-bit fixed-point representation
 6// (e.g., 2 bits for the integer part, 2 bits for the fractional part).
 7module ain_top (input wire clk,
 8                input wire rst,
 9                input wire signed [3:0] x1,
10                input wire signed [3:0] x2,
11                input wire signed [3:0] w1,
12                input wire signed [3:0] w2,
13                output reg signed [4:0] output_val); // Output size is larger to accommodate summation result
14
15   // Internal wires for intermediate calculations
16   wire signed [7:0] prod1, prod2;
17   wire signed [8:0] sum;
18   wire signed [4:0] activated_sum;
19
20   // Multiplication (can use dedicated DSP slices on an FPGA)
21   assign prod1 = x1 * w1;
22   assign prod2 = x2 * w2;
23
24   // Summation
25   assign sum = prod1 + prod2;
26
27   // Activation Function (Simple ReLU: if sum > 0, output = sum; else output = 0)
28   // We adjust sum back to a 5-bit width for a simple ReLU output.
29   assign activated_sum = (sum > 0) ? sum[4:0] : 5'b00000;
30
31   // Sequential logic to register the output on the clock edge
32   always @(posedge clk or posedge rst) begin
33      if (rst) begin
34         output_val <= 5'b00000;
35      end else begin
36         output_val <= activated_sum;
37      end
38   end
39
40endmodule

Directed Tests

AIN Testbench Source Code
 1//----------------------------------------------------------------------------------------------------------------------
 2// Artificial Intelligence Neuron (AIN) Testbench (TB) SystemVerilog Module
 3//----------------------------------------------------------------------------------------------------------------------
 4module ain_tb;
 5
 6   // Inputs are reg in testbench as we need to control their values over time
 7   reg clk;
 8   reg rst;
 9   reg signed [3:0] tb_x1, tb_x2, tb_w1, tb_w2;
10   // Output is wire in testbench to monitor the DUT's output
11   wire signed [4:0] tb_output_val;
12
13   // Instantiate the Design Under Test (DUT)
14   ain_top ain_top (.clk(clk),
15                    .rst(rst),
16                    .x1(tb_x1),
17                    .x2(tb_x2),
18                    .w1(tb_w1),
19                    .w2(tb_w2),
20                    .output_val(tb_output_val));
21
22   // Clock generation block
23   initial begin
24      clk = 0;
25      forever #5 clk = ~clk; // Clock with a period of 10 time units
26   end
27
28   // Stimulus and checking block
29   initial begin
30      // 1. Initialize inputs and apply reset
31      rst = 1;
32      tb_x1 = 4'd0;
33      tb_x2 = 4'd0;
34      tb_w1 = 4'd0;
35      tb_w2 = 4'd0;
36      #10; // Wait a bit
37      rst = 0; // Release reset
38      #10;
39
40      // 2. Apply test case 1 (Positive result: sum > 0)
41      // x1=2, w1=1 (prod1=2)
42      // x2=3, w2=2 (prod2=6)
43      // sum=8. ReLU output=8.
44      tb_x1 = 4'd2;
45      tb_w1 = 4'd1;
46      tb_x2 = 4'd3;
47      tb_w2 = 4'd2;
48      #10; // Wait for the next clock edge to register the output
49
50      // Check the output after a full clock cycle
51      if (tb_output_val == 5'd8) begin
52         $display("Test Case 1 Passed: Output is %d", tb_output_val);
53      end else begin
54         $display("Test Case 1 Failed: Expected 8, got %d", tb_output_val);
55      end
56
57      // 3. Apply test case 2 (Negative result: sum <= 0)
58      // x1=1, w1=1 (prod1=1)
59      // x2=3, w2=-1 (prod2=-3)
60      // sum=-2. ReLU output=0.
61      tb_x1 = 4'd1;
62      tb_w1 = 4'd1;
63      tb_x2 = 4'd3;
64      tb_w2 = -4'd1; // Represents -1 in 4-bit signed
65      #10;
66
67      // Check the output
68      if (tb_output_val == 5'd0) begin
69         $display("Test Case 2 Passed: Output is %d", tb_output_val);
70      end else begin
71         $display("Test Case 2 Failed: Expected 0, got %d", tb_output_val);
72      end
73
74      // 4. End simulation
75      $display("Simulation Finished");
76      $finish;
77   end
78
79   // Dump waveform data for visual inspection
80   initial begin
81      $dumpfile("dump.vcd");
82      $dumpvars(0, tb_top);
83   end
84
85endmodule

Universal Verification Methodology (UVM)

The Universal Verification Methodology (UVM) is a standardized methodology for verifying integrated circuit designs. It is based on the IEEE 1800.2 standard and provides a modular, object-oriented framework.

Key Benefits

  • Reusability: Standardized components can be reused across different projects and levels of design.

  • Automation: Features like factory overrides and built-in phases automate testbench construction.

  • Interoperability: Supported by all major EDA vendors.

Core Components

A typical UVM testbench consists of the following components:

UVM Driver

Converts transaction-level stimulus into pin-level activity for the Design Under Test (DUT).

UVM Monitor

Samples pin-level activity and converts it back into transactions for analysis.

UVM Sequencer

Manages the flow of transactions (sequence items) from sequences to the driver.

UVM Agent

A container that groups the sequencer, driver, and monitor for a specific interface.

UVM Scoreboard

Checks the correctness of the DUT by comparing actual output against expected results.

Simulation Phases

UVM components use standard execution phases to ensure consistent behavior across the testbench:

  1. build_phase: Instantiate and configure components.

  2. connect_phase: Establish communication links between components.

  3. run_phase: Execute the main simulation stimulus (time-consuming).

  4. report_phase: Summarize and display simulation results.
